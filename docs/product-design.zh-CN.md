# AACC 产品设计

[English version](product-design.md) · [仓库概览](../README.zh-CN.md)

## 产品承诺

AACC 为开发者提供一个持续可见、安静且本机运行的面板，只展示用户明确选择的 AI 编程任务。它让可行动的状态更容易被发现，但不会假装掌握本机集成无法可靠获得的信息。

## 核心用户

- 同时运行多个本地 AI 编程对话、需要快速注意运行、等待、完成或失败状态的开发者。
- 希望通过本地 API 或 CLI 接入自定义 Agent Hook、但不想把任务数据交给云端控制台的高级用户。
- 需要保守统一状态模型、而非与 GUI 强耦合方案的 Adapter 贡献者。

## 体验原则

1. **先选择，后观察。** AACC 只展示和轮询用户选择的任务。
2. **一眼可读，避免噪音。** 大尺寸状态灯、耗时、任务标题和简短原因让下一步判断清晰可见。
3. **控制必须明确。** 选择卡片不能意外隐藏面板或切换焦点；聚焦和输入必须是显式操作。
4. **诚实表达不确定性。** 缺失或模糊的信号保持 `UNKNOWN` 或 `WARNING`，绝不静默伪装成“已完成”。
5. **默认本机化。** 配置和历史保留在本机，API 仅绑定 loopback，不上传对话。

## 状态模型

状态管理器涵盖未配置、活动、等待、终态、告警、错误、暂停、取消、停止和未知状态。每次更新都带有来源和可信度。新鲜高可信度信息不会被弱信号误覆盖；新的任务启动事件可以让已终结任务重新进入活动态。

对于 Codex，会话事件具有明确语义：`task_started` 表示当前轮次活跃，`task_complete` 表示当前轮次完成。这些事件的优先级高于单纯的文件更新时间。

## 数据流

```text
本机 Codex 索引 / 已选会话事件 / Agent Hook / CLI 包装器
                                ↓
                         任务发现和 Adapter
                                ↓
              含来源与可信度的标准化 TaskState
                                ↓
                   SQLite 历史与观察者通知
                                ↓
          PySide6 悬浮面板、菜单栏、localhost API、CLI
```

Codex 发现适配器只读取判断状态所需的已选会话元数据：标识、标题、时间戳、事件标签和进程信息；不解析提示词正文、代码或命令文本。

## 交互模型

- 悬浮面板记住位置，并允许用户自主选择是否始终置顶。
- 设置提供任务筛选、透明度与位置控制。
- 单击卡片仅选中任务；显式的右键菜单才提供聚焦操作。
- 紧凑模式降低视觉密度；完整面板显示状态原因和任务详情。
- 菜单栏入口让隐藏后的面板始终可找回。

## 安全与隐私边界

API 只绑定 loopback 并使用 Token 保护，不接收任意 shell 命令。键盘注入仅允许白名单按键，且必须先成功聚焦配置的目标。日志会脱敏常见密钥模式。对没有公开可靠状态接口的第三方 Agent，使用配置、包装器或手动状态降级，而不伪造精确度。

## V1.0 非目标

AACC 不是云同步服务、远程控制平台、移动应用、屏幕读取工具、Token 成本估算器或多 Agent 工作流调度器。它不会绕过 Agent 审批或 macOS 权限。

## 成功标准

用户可以选择少量本机任务、将 AACC 固定在稳定桌面位置、立即注意到状态变化，并理解应用何时无法确认状态。贡献者可以添加 Adapter，而无需修改 GUI 或削弱本机安全边界。
